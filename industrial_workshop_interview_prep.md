# 工业车间环境保障系统面试准备

## 1. 项目一句话介绍

这是一个基于 `STM32F407ZGT6 + FreeRTOS` 的工业车间环境保障系统，我主要负责硬件联调和嵌入式软件实现。系统通过 `Modbus RTU` 采集温湿度、PM2.5/PM10、气体等有线传感器数据，通过 `ESP01S` 接入网络并用 `MQTT` 和网页联动，同时接入两路 `ESP32` 无线节点上报门磁、光照和人体存在信息，最后结合本地串口屏、继电器和 PWM 风扇实现本地自动控制、远程控制和参数持久化。

## 2. 面试开场可以怎么讲

### 2.1 30 秒版本

我做的是一个车间环境保障系统，核心 MCU 是 `STM32F407`，软件架构用 `FreeRTOS`。系统一边通过 `Modbus RTU` 轮询温湿度、PM、气体传感器和继电器板，一边通过 `ESP01S` 把数据发到 `MQTT`，并接收网页端下发的控制命令。同时我还接了两个 `ESP32` 无线节点，用 `UDP + JSON` 上报门磁、光照和人体状态。控制层会根据阈值做加热、制冷、加湿、照明、报警和风扇调速，参数还能存到片内 `Flash`，断电后恢复。

### 2.2 1 分钟版本

这个项目我重点做了三层。第一层是采集层，STM32 通过 `UART1` 做 `Modbus RTU` 主站，轮询温湿度、PM2.5/PM10、气体传感器，并读取继电器板状态；同时通过 `UART2` 接 `ESP01S`，接收两个 `ESP32` 节点用 UDP 发来的无线 JSON 数据。第二层是控制层，我把温度、湿度、PM、气体、门磁、光照和人体状态统一汇总成一个线程安全的数据快照，然后在控制任务里做阈值判断、迟滞控制、报警延时、手自动切换和风扇多档调速。第三层是交互层，本地有串口屏做分页显示、阈值设置和手动控制，远程通过 `MQTT` 和网页交互，所有关键阈值还能做 `Flash` 持久化。整个系统重点解决的是多串口并发、协议解析、自动控制稳定性以及掉线恢复。

### 2.3 3 分钟版本

这个系统本质上是一个“多源数据采集 + 本地闭环控制 + 远程可视化与控制”的完整方案。

硬件上，`STM32F407` 是主控。`UART1` 接 Modbus 传感器和继电器板，`UART2` 接 `ESP01S` 做 WiFi/MQTT/UDP，`UART4` 接串口屏，另外风扇通过 `TIM3 PWM` 调速。无线侧我用了两个 `ESP32` 节点，一个负责门磁，另一个负责光照和人体存在检测，它们通过 UDP 把 JSON 数据发给 STM32 网关。

软件上，我用 `FreeRTOS` 把系统拆成几个任务：控制任务负责周期采集和执行控制逻辑，MQTT 任务负责联网、发包、收包和远程命令，串口屏任务负责本地人机交互，显示任务负责 OLED 页面轮播，诊断任务负责统计堆、最小剩余堆和各任务栈水位。这样做的好处是职责边界清晰，也便于后续定位问题。

协议实现上，Modbus 这块我做了 CRC16 校验、响应长度判断、异常帧判断和超时重试。ESP01S 这块我没有直接用 AT 自带的 MQTT，而是在 AT TCP 连接基础上自己做了 MQTT 3.1.1 的报文封装和解析，包括 `CONNECT / CONNACK / SUBSCRIBE / SUBACK / PUBLISH / PINGRESP`，并且处理了 `+IPD` 的多种格式和 `CIPMUX` 多链路场景。为了避免 MQTT 和 UDP 共用一个串口时互相抢数据，我还做了一个 `pending IPD` 缓冲队列，把不属于当前链路的数据先缓存起来，后面再取。

控制策略上，我不是简单做“超过阈值就开，低于阈值就关”，而是加了迟滞、最小切换间隔和报警延时，这样能避免继电器频繁抖动。风扇速度是多来源合成出来的，既考虑除湿需求，也考虑 PM 和气体浓度，最后取需求里最高的档位。为了方便现场调试，我把阈值设置、手动控制、恢复出厂、动作记录都接到了串口屏，同时远程网页也可以改参数。参数变更后不会立刻频繁写 Flash，而是做了 1 秒去抖，并且通过 `magic + version + CRC32` 保证参数有效性。

这个项目里我觉得最有代表性的难点有三个：第一，多串口和多协议并发下的数据解析和边界控制；第二，本地自动控制和远程手动控制怎么共存；第三，网络抖动、串口分包和异常场景下系统如何保持稳定。

## 3. 系统架构怎么讲

### 3.1 文字版架构图

`Modbus 传感器/继电器板 -> UART1 -> STM32 控制任务 -> 控制逻辑 -> 继电器/PWM 风扇`

`ESP32 门磁节点 -> UDP(JSON) -> ESP01S -> UART2 -> STM32 -> 数据汇总/状态判断`

`ESP32 光照人体节点 -> UDP(JSON) -> ESP01S -> UART2 -> STM32 -> 数据汇总/状态判断`

`STM32 -> MQTT -> 云端 Broker -> 网页前端/后台`

`串口屏 <-> UART4 <-> STM32`

`参数阈值 -> STM32 内部 Flash 持久化`

### 3.2 FreeRTOS 任务划分

- `ControlTask`
  负责周期刷新 Modbus 传感器数据、获取数据快照、运行自动控制逻辑、触发参数持久化处理。
- `MqttApp_Task`
  负责 ESP01S 初始化、WiFi 接入、MQTT 连接、命令接收、遥测上报、UDP 无线节点接入。
- `Screen_Task`
  负责串口屏收包解析、页面刷新、阈值下发、手动控制、恢复出厂、动作记录显示。
- `DisplayCycleTask`
  负责 OLED 轮播显示传感器值和继电器状态。
- `Task_PC13`
  负责心跳灯，便于看系统是否还在正常运行。
- `Diag_Task`
  负责堆、最小剩余堆、任务栈水位、网络状态诊断输出。

### 3.3 各串口分工

- `UART1`
  Modbus RTU 主站，采集温湿度、PM、气体传感器并控制继电器板。
- `UART2`
  ESP01S 通信口，承载 WiFi、MQTT、UDP 多协议数据。
- `UART3`
  主要用于调试日志打印。
- `UART4`
  串口屏交互。

## 4. 核心功能是怎么实现的

### 4.1 Modbus RTU 采集和解析

#### 你可以这样说

我在 STM32 上实现的是 `Modbus RTU 主站`。发送侧由主站主动轮询各个从站，比如温湿度传感器、PM 传感器、气体传感器和继电器板。接收侧不是在中断里直接做完整解析，而是采用“`中断收字节 + 任务态取快照并解析`”的方式。

#### 具体实现细节

1. `USART1` 中断每收到一个字节，就先放进 ISR 缓冲区。
2. 当收到功能码后，根据功能码初步判断一帧应该有多长。
3. 对于 `0x03 / 0x04` 这种读寄存器响应帧，前几个字节里会带 `byte count`，我会再动态算出完整帧长。
4. 当收到足够长度后，先标记“本帧完成”，但不在中断里做复杂解析。
5. 任务态读取时会短暂关中断，把 ISR 缓冲区拷贝到快照缓冲区，避免任务和中断同时读写同一份数据。
6. 之后再在任务里做 CRC16、从站地址、功能码、字节数、异常码等完整校验。
7. 校验通过后再按协议字段解析寄存器数据，比如温湿度寄存器、PM2.5/PM10 寄存器和气体值寄存器。

#### 为什么这样做

- 中断里只做“短平快”的收字节动作，避免阻塞。
- 协议解析、CRC 计算、异常判断放到任务里，实时性和可维护性更好。
- 通过快照复制避免竞争条件。

### 4.2 收到的 Modbus 数据是怎么解析的

这是很容易被问到的一题，建议你直接按下面的逻辑回答。

#### 回答模板

我收到 Modbus 响应后，第一步不是直接取值，而是先做完整性判断。先看帧长够不够，再做 `CRC16` 校验，然后检查从站地址和功能码是不是我当前发出去的命令对应的那一个。如果功能码最高位被置位，说明这是异常响应，我会按异常帧处理。对于 `0x03 / 0x04` 的读寄存器响应，我还会校验 `byte count` 是否和实际数据区一致。

全部校验通过后，才会去按寄存器格式解析。比如温湿度传感器的响应里，前两个寄存器分别是湿度和温度，我会把高低字节拼成 `uint16_t`，再除以 10 得到实际值。温度我还按 `int16_t` 去解释，这样负温场景也能兼容。PM 传感器我解析出 `PM2.5` 和 `PM10`，气体传感器则解析成对应浓度值。这样可以保证上层控制逻辑拿到的数据一定是经过校验的有效数据。

### 4.3 为什么 Modbus 总线要加互斥锁

因为同一时刻只有一个主站事务能在总线上进行。如果多个任务同时往 `UART1` 发 Modbus 指令，会导致请求和响应交叉，后面的解析就会对不上。我这里是在 Modbus 发送函数外层加了 `Mutex`，保证一次完整的“发送请求 -> 等待响应 -> 校验解析 -> 超时/重试”是原子的，这样总线访问是串行的，不会互相干扰。

### 4.4 传感器数据如何统一管理

我没有让控制逻辑直接散着访问每个模块，而是设计了一个统一的 `AllSensorData` 数据中心。它的作用有三个：

1. 统一保存温湿度、PM、气体、门磁、光照、人体存在这些数据。
2. 给每类数据打上 `valid bit` 和时间戳。
3. 对外提供线程安全的快照接口。

这样控制任务拿到的是一份完整快照，而不是边读边变的分散变量。无线节点掉线或者某类传感器暂时无效时，也能明确知道是哪一类数据失效了。

### 4.5 无线节点的数据是怎么接入的

两个 ESP32 节点通过 WiFi 连上同一个网络，采用 `UDP + JSON` 往网关发数据。STM32 这边通过 `ESP01S` 开一个 UDP 本地端口监听，当收到 JSON 后先识别 `type` 字段，再按不同类型更新本地数据中心。

#### 三类典型数据

- `door`
  包含门磁闭合状态和电平。
- `human`
  包含人体存在状态和距离。
- `light`
  包含光照强度 lux。

#### 为什么选 UDP

- 传感器上报是短报文、周期性、允许个别丢包。
- UDP 开销小，节点实现简单。
- 对实时状态上报很合适。

### 4.6 JSON 是怎么解析的

我这里没有上重量级 JSON 库，而是自己做了一个轻量级平面 JSON 解析器，思路是通过字符串搜索定位 key，再找到冒号后的值，然后按 `int / float / bool / string` 分别解析。这个解析器不打算覆盖完整 JSON 标准，比如不支持复杂嵌套对象和数组，但对于我这个项目里固定格式的短 JSON 报文足够了。

#### 面试时推荐补一句

这个取舍主要是基于 MCU 资源和协议可控性考虑。因为无线节点的 payload 是我自己定义的，字段很固定，所以用轻量解析更省资源、更容易控时延。如果以后协议复杂度上升，我会换成更完整的 JSON 库或者改为二进制协议。

### 4.7 ESP01S 这块是怎么做的

#### 可以这样概括

ESP01S 在这个项目里承担的是“网络协处理器”的角色。STM32 通过 AT 指令控制 ESP01S 入网、建立 UDP 和 TCP 连接，但 MQTT 报文的封装和解析是由 STM32 侧自己完成的。

#### 为什么不用 AT 自带 MQTT

我没有直接用 AT 固件封装好的 MQTT 指令，而是选择在 TCP 连接上自己做 MQTT 3.1.1 报文。原因有三点：

1. 可控性更强，可以自己管理 topic、重连、重发、ACK 节奏。
2. 兼容性更好，不被某个 AT 固件版本绑死。
3. 更容易同时兼顾 UDP 和 MQTT 两种业务。

### 4.8 MQTT 报文是怎么实现的

我在 STM32 侧自己实现了 MQTT 3.1.1 的基础报文编码和解析，主要包括：

- `CONNECT`
- `CONNACK`
- `SUBSCRIBE`
- `SUBACK`
- `PUBLISH`
- `PINGREQ`
- `PINGRESP`

像 MQTT 的 `Remaining Length` 是变长编码，我这里也按协议自己做了编解码。收到 MQTT 数据包之后，会先解析固定头，再解析变长剩余长度，然后根据包类型分发处理。命令类 topic 会转成控制动作，状态类 topic 会用于时间同步或网页在线状态维护。

### 4.9 为什么 ESP01S 那边要做 pending IPD 队列

这个点非常值得讲，因为它能体现你对多协议复用的理解。

ESP01S 开了 `CIPMUX=1` 多链路之后，同一个 UART 接收流里会混着 `UDP` 和 `MQTT(TCP)` 两种链路的数据。假如我当前在处理 MQTT 链路，但串口里先来了一个 UDP 的 `+IPD`，如果直接丢弃，数据就没了；如果强行当成 MQTT 处理，也会解析错。

所以我的做法是：

1. 先解析 `+IPD` 头部，拿到 `link id`、数据长度、可选的 IP 和端口。
2. 如果这帧属于当前想处理的链路，就直接取出来处理。
3. 如果不属于当前链路，就先缓存到 `pending IPD` 队列。
4. 后面轮到那个链路处理时，再优先从 pending 队列里取。

这样就实现了“一个串口、多条链路、多种协议”的安全复用。

### 4.10 远程命令如何避免重复执行

网页端下发命令时会带 `req_id`。STM32 这边维护了一个 ACK 缓存，如果收到相同 `req_id` 的重复命令，就不再重复执行实际动作，而是直接重放之前缓存的 ACK 响应。这样做的意义是：

- 网络抖动导致网页重发时，不会重复开关继电器。
- 可以保证远程控制是幂等的。
- 也方便前端做超时重试。

### 4.11 为什么还要做 ACK 延迟队列

MQTT 收包轮询时如果边收边立即发 ACK，有可能打乱串口接收节奏，尤其是 ESP01S 同时还在处理 `+IPD` 和 MQTT 数据的时候。所以我把 ACK 做成了“延迟发送队列”：

1. 在集中处理 MQTT 收包的阶段，先只把 ACK 放进队列。
2. 等收包阶段结束后，再批量尝试发送。

这样可以减少串口拥塞，也能降低在协议解析过程中插入额外发送造成的问题。

### 4.12 自动控制逻辑是怎么设计的

控制逻辑不是简单的 if-else，而是分成几个独立模块：

- 温度控制
  决定加热和制冷。
- 湿度控制
  决定加湿，以及把除湿需求交给风扇。
- 光照控制
  结合人体存在和环境光决定灯光。
- 报警控制
  结合门磁超时、气体超阈值、PM 超阈值决定声光报警。
- 风扇控制
  综合除湿、PM、气体三类需求，选择最高档位。

### 4.13 为什么要加迟滞和最小切换时间

如果只按单一阈值判断，比如温度大于 30 度开制冷，小于 30 度关制冷，那么温度在 30 度附近波动时继电器会频繁吸合释放，既不稳定，也影响寿命。

所以我做了两层保护：

1. `迟滞`
   比如高温阈值是 30，迟滞是 1，那么超过 30 才开，但要低于 29 才关。
2. `最小切换间隔`
   即使判断条件来回变化，也要求距离上次切换已经过了一段最短时间，才允许再次动作。

### 4.14 风扇为什么是多来源综合控制

因为风扇在项目里不只是通风，还承担除湿辅助功能。所以我不是只看一个传感器，而是把三类需求综合起来：

1. 湿度高时，需要通风除湿。
2. PM2.5 高时，需要增加换气强度。
3. 气体浓度高时，也需要更高风量。

这三路需求各自映射成档位，最终取最高档作为当前风扇档位。这样做的好处是控制逻辑更符合实际场景，不会出现某个污染指标很高但风扇却被别的条件压低的情况。

### 4.15 手动控制和自动控制是怎么共存的

我对每个设备都做了一个手动覆盖结构，里面记录：

- 是否处于手动模式
- 手动强制值
- 手动开始时间
- 手动持续时长

控制逻辑在执行时会先检查该设备有没有有效的手动覆盖。如果有，就优先执行手动值；如果超时了，就自动恢复到自动模式。这样既满足现场调试和人工干预，也不会因为忘记切回自动而让系统长期跑偏。

### 4.16 串口屏是怎么通信的

串口屏部分我没有依赖复杂协议，而是自定义了一个非常轻量的二进制帧格式，帧头是 `0x55`，帧尾是 `0xAA`。不同功能对应不同帧长：

- 3 字节帧
  主要用于页面切换。
- 7 字节帧
  用于阈值设置和自动恢复。
- 8 字节帧
  用于手动控制和恢复出厂。

串口本身是字节流，存在分包和粘包，所以我用了滑动窗口去识别这些固定长度帧。每收到一个新字节，就更新窗口，然后按优先级去匹配不同类型的命令帧。这样即使串口数据不是整帧一次到达，也能稳定识别。

### 4.17 为什么串口屏做了文本缓存去重

因为很多页面内容是周期刷新，如果每次都把同样的文本重复下发给串口屏，会增加串口带宽占用，也会让界面刷新显得抖动。所以我做了一个文本缓存，只有内容真的发生变化时才重新发送。这个优化虽然简单，但对串口屏这类低带宽外设非常实用。

### 4.18 参数持久化是怎么做的

参数持久化存在 STM32 片内 `Flash` 的固定扇区里。我存的不是零散变量，而是一个完整配置块，里面包括：

- `magic`
  用来识别这是不是我自己的配置数据。
- `version`
  用来做结构版本兼容。
- 阈值结构体
- 风扇档位表
- `CRC32`
  用来校验整块数据是否损坏。

启动时先读出整块数据，检查 `magic`、`version`、风扇表合法性和 `CRC32`。如果任何一步失败，就回退到出厂参数，并重新保存一份。

### 4.19 为什么写 Flash 要做 1 秒去抖

参数在串口屏和网页上都可能被连续修改，如果每改一次就立刻擦写 Flash，会带来两个问题：

1. Flash 擦写寿命消耗快。
2. 多次快速修改会产生很多无意义写入。

所以我的做法是：参数变更时先标记“待保存”，记录版本号和时间戳；控制任务周期性调用持久化处理函数，只有在一段时间内没有继续变化时，才真正写入 Flash。这样可以明显减少写入次数。

### 4.20 为什么写 Flash 时要暂停调度

因为片内 Flash 擦除和编程期间对实时性和代码执行有影响。为了避免其它任务在关键写入窗口里打断流程，我在真正擦写时做了短时间的调度保护，保证这次配置块写入是连续且一致的。

### 4.21 在线/离线状态是怎么判断的

我不是单纯看有没有收到过数据，而是给每类无线传感器都记录最近一次更新时间。如果当前时间减去上次收到数据的时间超过设定阈值，就认为该节点离线。继电器链路则是通过轮询读取状态，连续失败达到次数后才标记离线。这样比一次超时就判死更稳健。

### 4.22 诊断任务做了什么

诊断任务会定时输出：

- 当前剩余堆
- 历史最小剩余堆
- 各任务栈高水位
- WiFi、MQTT、网页在线状态
- ESP01S 最近一次错误码

这个模块的意义是，一方面支撑开发阶段排查内存和栈风险，另一方面也能和简历里写的 `heap/min-heap/HighWaterMark` 对上。

## 5. 简历上这几个点，面试时可以怎么展开

### 5.1 FreeRTOS 架构

#### 面试回答

我把系统按职责拆成了控制、MQTT 通信、串口屏、显示、心跳和诊断几个任务，而不是所有逻辑都堆在 `while(1)` 里。这样做的好处是功能边界清晰，串口通信和控制逻辑互不阻塞。同时我还加了栈溢出 Hook，并定期输出 heap、min-heap 和各任务的 HighWaterMark，用来定位任务栈和内存是否足够。

### 5.2 ESP01S 网络通信

#### 面试回答

ESP01S 这块我重点做了三件事：第一是 AT 层初始化和异常恢复，包括自动波特率探测、退出透传、入网和多连接配置；第二是 `+IPD` 数据解析，兼容不同格式并支持 `CIPMUX` 多链路；第三是在 TCP 之上自己实现 MQTT 3.1.1 的基础封包和解包。为了避免 UDP 和 MQTT 共用同一串口时互相抢数据，我还加了 pending 队列和 ACK 延迟发送队列，保证通信稳定性。

### 5.3 多串口资源管理

#### 面试回答

项目里一共管理了 `UART1/2/3/4` 四路串口，分别负责 Modbus、ESP01S、调试日志和串口屏。我的处理思路是按业务特性区分：Modbus 这种请求-响应型协议用中断收包加任务解析；ESP01S 这种多协议混流场景用环形缓冲和链路分发；串口屏用滑动窗口识别固定格式帧。这样不同串口的接收策略和上层协议特点是匹配的。

### 5.4 串口屏交互

#### 面试回答

串口屏部分我自定义了一套轻量协议，把页面切换、阈值设置、手动控制和恢复出厂分成不同长度的二进制帧。接收时使用滑动窗口识别，发送时加入文本缓存去重，避免重复刷新。这样本地屏既能实时显示数据，也能直接修改阈值和控制设备。

### 5.5 ModbusRTU 主站驱动

#### 面试回答

Modbus 这块我做的是主站轮询驱动，支持 CRC16 校验、功能码识别、异常帧处理、超时重试和从站数据解析。接收上采用 ISR 收字节、任务态取快照并校验解析的方式，同时在总线访问外层加 Mutex，避免多个任务并发访问串口导致帧混乱。

### 5.6 参数持久化

#### 面试回答

参数持久化我不是简单直接写变量，而是设计了一个带 `magic`、`version` 和 `CRC32` 的配置块，存到片内 Flash 固定扇区。启动时先校验，失败就恢复出厂参数；运行时对参数修改做 1 秒去抖，再统一写入，减少 Flash 擦写次数并保证数据一致性。

## 6. 高频面试题和参考答案

下面这些问题，基本都是结合你的代码实现整理的。回答我尽量写成“你可以直接说”的口吻。

### Q1：你这个项目的核心价值是什么？

答：核心价值是把多个环境传感器、无线节点、本地执行器和远程网页控制整合成一个闭环系统。不是单纯采集数据，而是能根据环境状态自动做温湿度、空气质量、照明和报警联动，同时还支持本地屏和网页双端交互。

### Q2：为什么用 STM32F407？

答：这个项目外设比较多，需要多路串口、定时器 PWM、足够的 Flash 和 RAM，F407 在资源上更从容，做 FreeRTOS 多任务也更稳妥。像我这里就同时用了多路 UART、PWM、片内 Flash 持久化和多个协议栈，F407 比较合适。

### Q3：为什么用 FreeRTOS，而不是裸机？

答：因为这个项目不是单一循环采集，而是 Modbus、ESP01S、串口屏、显示、诊断这些任务并行存在。用 RTOS 以后，各模块职责更清晰，也更容易控制优先级和时序关系。尤其是通信和控制并存的场景，RTOS 会比裸机状态机更好维护。

### Q4：你的任务是怎么划分的？

答：控制任务负责采集和自动控制，MQTT 任务负责联网和远程通信，串口屏任务负责本地人机交互，显示任务负责 OLED 轮播，诊断任务负责资源监控，另有一个心跳灯任务。这样能把实时控制、通信和界面分开，互不阻塞。

### Q5：收到 Modbus 数据后，具体是怎么判断一帧完整的？

答：我在接收中断里按字节收数据。收到功能码后，先根据功能码判断基础帧类型，比如 `0x06` 的响应固定就是 8 字节；如果是 `0x03/0x04` 读寄存器响应，等收到 `byte count` 后再动态计算总长度，达到长度后才认为一帧完成。

### Q6：为什么不在中断里直接解析 Modbus？

答：因为中断里应该尽量短，只做收字节和状态标记。CRC 校验、异常帧处理、寄存器拼接这些都放到任务里做，更安全也更容易维护。如果中断里做太多事情，会拖长中断时间，影响系统实时性。

### Q7：你的 CRC16 是怎么做的？

答：按 Modbus 标准 CRC16 做的，多项式是 `0xA001`。我对数据区逐字节异或，再右移 8 次并按最低位是否为 1 决定是否异或多项式，最后得到 16 位 CRC，发送时低字节在前、高字节在后。

### Q8：如果 Modbus 收到异常帧，你怎么处理？

答：我会先看功能码最高位是不是被置位，如果是，说明从站返回的是异常响应，就不会再取数据，而是按失败处理。上层会根据失败结果做重试或者标记该次采集无效。

### Q9：为什么要给 Modbus 操作加互斥锁？

答：因为总线是共享的，一次完整事务是“发请求 + 等响应 + 校验解析”。如果两个任务同时发命令，请求和响应会串掉，所以必须用 Mutex 保护这一整段事务。

### Q10：温湿度、PM、气体这些数据是怎么交给控制层的？

答：我统一放进一个线程安全的数据结构里，并提供快照接口。控制层每个周期只拿一次快照，用这份稳定数据做决策。这样不会出现控制逻辑执行到一半，底层数据又被别的任务改掉的情况。

### Q11：无线节点为什么不用 MQTT，为什么用 UDP？

答：因为无线节点只需要短报文、周期性上报状态，对实时性要求高，对单包可靠性要求没那么高，用 UDP 更轻量。MQTT 放在网关到服务器这一层更合适，由 STM32 网关统一接入。

### Q12：你自写的 JSON 解析器有什么特点？

答：它是面向固定格式短报文的轻量解析器，支持 `bool/int/float/string`，实现简单、资源占用低，但不支持复杂嵌套和数组。这个设计是有意取舍，因为节点发来的 JSON 是我自己定义的，字段固定，不需要上完整通用库。

### Q13：ESP01S 的 `+IPD` 是怎么解析的？

答：我先匹配 `+IPD` 头，然后分情况解析有没有 `link id`、有没有源 IP 和端口，拿到长度后再截取后面的 payload。因为不同 AT 固件 `+IPD` 格式不完全一样，所以我兼容了几种常见形式。

### Q14：MQTT 为什么自己封包，不直接用 AT 的 MQTT 指令？

答：主要是为了可控性和兼容性。自己封包后，我可以自由定义 topic、重连逻辑、PING 节奏、ACK 策略，也不依赖某个特定版本固件的 MQTT 实现。对于这种需要同时处理 UDP 和 MQTT 的项目，自己掌握报文层会更灵活。

### Q15：你实现了 MQTT 的哪些部分？

答：实现了最基本但够用的一套，包括 `CONNECT/CONNACK`、`SUBSCRIBE/SUBACK`、`PUBLISH` 和 `PINGREQ/PINGRESP`，同时实现了 MQTT 的变长长度字段编解码。这样能完成命令订阅、遥测上报和保活。

### Q16：如果 WiFi 或 MQTT 断开了怎么办？

答：我做了分层恢复。先检测 ESP01S 当前是否还能正常 AT 通信，再检查 WiFi 是否拿到 IP，然后检查 MQTT TCP 链路和 PING/消息收发是否正常。某一层异常时，就从对应层开始重建，必要时重新初始化 ESP01S。这样恢复更快，也不会每次都粗暴整机重启。

### Q17：为什么还要维护网页在线状态？

答：因为有些控制策略或展示逻辑要知道远程端是不是还在活跃，比如状态上报节奏、界面联动、命令响应等。这里我通过命令接收和时间窗口维护一个 web alive 状态。

### Q18：远程命令怎么防止重复执行？

答：通过 `req_id + ACK 缓存` 做幂等。如果命令重复到达，我不重复执行动作，只重发之前的 ACK。这样不会因为网络重试导致继电器被重复开关。

### Q19：为什么要做 ACK 延迟队列？

答：因为集中收 MQTT 包时如果立刻同步发 ACK，可能打乱当前串口接收节奏，尤其在 ESP01S 多链路混流时风险更高。所以我把 ACK 先排队，等接收处理阶段结束后再发，提高整体稳定性。

### Q20：自动控制中，温度控制是怎么实现的？

答：温度控制分高低阈值和迟滞。低于低温阈值时开加热，高于“低温阈值 + 迟滞”才关加热；高于高温阈值时开制冷，低于“高温阈值 - 迟滞”才关制冷。这样避免边界抖动。

### Q21：湿度和风扇控制是怎么联动的？

答：湿度低时直接开加湿；湿度高时不单独开一个除湿器，而是把“除湿需求”转成风扇档位需求，和 PM、气体两个需求一起综合，最后选最高档位输出到 PWM 风扇。

### Q22：照明为什么同时参考人体和光照？

答：因为只看光照会出现“环境暗但没人也开灯”，只看人体会出现“有人但环境本来很亮也触发”的问题。结合人体存在距离和光照阈值，可以更接近实际使用需求。

### Q23：报警为什么要做延时？

答：像门磁打开、PM 升高、气体瞬时抖动这些场景，如果立刻报警，误报会比较多。所以我对门开超时、PM 超阈值、气体超阈值都做了持续时间判断，只有持续满足条件才触发报警。

### Q24：手动模式为什么要带超时？

答：因为现场调试时经常会临时强制开关设备，如果没有超时恢复，很容易忘记切回自动，系统就长期失去闭环控制。所以我做成了“手动覆盖一段时间，超时自动回归自动模式”。

### Q25：串口屏为什么用固定长度帧和滑动窗口？

答：因为串口本质是连续字节流，固定长度帧更容易做稳定解析，滑动窗口则能适应粘包和分包，不要求一次正好读到整帧。在 MCU 上这种方案很轻量，也足够可靠。

### Q26：为什么做文本缓存去重？

答：减少串口冗余发送，避免界面闪烁，也能降低 CPU 和串口带宽占用。对于周期刷新页面，这个优化很直接有效。

### Q27：参数持久化为什么不用 EEPROM？

答：这个板子本身就有足够的片内 Flash，而且参数量不大，所以直接用片内 Flash 更省硬件成本。为了保证可靠性，我加了版本号、CRC32、去抖写入和恢复出厂机制。

### Q28：为什么要做 magic、version 和 CRC32？

答：因为单纯把结构体写进去不够稳妥。`magic` 用来判断是不是我的配置块，`version` 用来兼容以后结构变化，`CRC32` 用来判断数据有没有损坏，这三层一起能大幅提高配置恢复的可靠性。

### Q29：你怎么减少 Flash 擦写次数？

答：两招。第一，参数修改不立即写，而是做延时去抖；第二，如果准备写入的新配置和当前 Flash 里完全一致，就跳过这次写入。这样能明显减少无效擦写。

### Q30：你怎么确认 FreeRTOS 任务栈有没有配够？

答：我加了栈溢出 Hook，同时通过 `uxTaskGetStackHighWaterMark()` 周期性查看各任务栈高水位，再结合实际运行压力去调整。这样不是拍脑袋配栈，而是有数据依据。

### Q31：这个项目里你觉得最难的点是什么？

答：我觉得最难的是 ESP01S 这部分的多协议复用。因为 UART2 上混着 AT 回显、`+IPD`、UDP payload 和 MQTT payload，链路还不止一条。如果没有链路分发和 pending 队列，数据非常容易串。这个问题解决以后，整个远程通信稳定性就上来了。

### Q32：你做过哪些稳定性处理？

答：主要有 CRC 校验、异常帧判断、超时重试、链路掉线重连、传感器在线超时判定、迟滞控制、最小切换间隔、ACK 去重、参数写入去抖、Flash CRC 校验和任务栈监控。

### Q33：这个项目里你有什么可以体现“工程化思维”的地方？

答：我觉得有几个点比较明显：一是把通信、控制、显示、诊断分层；二是通过数据快照和互斥锁处理并发；三是用迟滞、最小切换时间、报警延时提升控制稳定性；四是做了配置块校验、掉线恢复和资源监控，而不是只让功能跑通。

### Q34：如果让你再优化一次，你会优化什么？

答：我会从三方面继续优化。第一，ESP01S 串口接收可以升级成 `DMA + 空闲中断`，提升高负载下的吞吐和 CPU 利用率。第二，WiFi 配网和 Broker 参数可以做成串口屏可配置，而不是当前演示版里的固定配置。第三，可以给协议层增加更系统化的日志和状态机，把异常恢复流程再标准化。

### Q35：这个项目里有没有你主动做的优化？

答：有几个我觉得比较实用。比如 Modbus 接收用了“ISR 收包、任务态快照解析”，减少中断负担；ESP01S 做了 pending 队列和 ACK 延迟队列，提升多链路稳定性；串口屏加了文本缓存去重；参数持久化做了去抖和重复内容跳过写入。这些都不是必须功能，但对系统稳定性帮助很大。

### Q36：网页端和后端大概是怎么配合的？

答：网页端实时数据主要直接通过 `MQTT over WebSocket` 订阅，历史数据则由本地 `Node.js + SQLite` 服务存储和提供查询接口。也就是说，实时链路和历史链路是分开的，实时看 MQTT，追溯看数据库，这样结构比较清晰。

### Q37：如果面试官问“你这个项目有没有不足”，怎么答？

答：可以坦诚说有。比如当前演示版本里 WiFi 和 Broker 参数还是固定写在代码里的，适合开发调试，但如果走产品化，我会把它们也做成可配置并加权限管理。再比如无线节点到网关用的是 UDP，它适合状态上报，但如果以后有更强的可靠性要求，可以在应用层再加确认和重发机制。

### Q38：如果面试官问“你在项目里最能体现能力的一块是什么”，怎么答？

答：我会说是“多协议通信和控制闭环的整合能力”。因为这个项目不是单点技术，而是 Modbus、UART、UDP、MQTT、串口屏、Flash 持久化、FreeRTOS 调度和现场控制逻辑都串起来了，而且这些模块之间是有协同关系的。

## 7. 面试官很可能会继续追问的细节

### 7.1 “门磁、光照、人体这三类无线数据最后怎么参与控制？”

答：门磁主要参与报警逻辑，持续开门超过设定时长才触发报警；光照和人体一起参与照明逻辑，有人且距离在范围内或者环境亮度低于阈值时会开灯；人体和光照数据都会进入统一数据中心，再由控制任务周期性判断。

### 7.2 “继电器状态为什么还要读回来？”

答：因为不能只相信软件里的目标状态。实际现场可能有通信失败、继电器板未响应、继电器板状态被别的命令改变等情况，所以我会周期性读回继电器状态，用于显示、状态上报和链路健康判断。

### 7.3 “温度是负数怎么办？”

答：温度寄存器解析时我按 `int16_t` 处理，而不是单纯 `uint16_t`，这样负温值也能正确解释。

### 7.4 “如果某一个传感器读失败，会不会影响整个系统？”

答：不会。每类传感器都有自己的有效位和时间戳，控制层用快照时会知道哪些数据有效、哪些暂时无效。某一路失败不会把整个数据结构搞乱，只会让依赖那一路的控制逻辑进入保守处理。

### 7.5 “恢复出厂做了什么？”

答：恢复出厂时会把阈值恢复成预设的默认值，清掉手动控制状态，然后请求重新写入 Flash。为了防误触，我还对恢复出厂命令做了独立识别和一定的去抖限制。

## 8. 可以主动强调的项目亮点

如果面试官没有主动问，你可以自己挑 2 到 3 个说，不要全说完。

1. 通信层不是只会“调库”，而是自己实现了 Modbus 主站、`+IPD` 解析和 MQTT 基础报文。
2. 多串口并发场景下做了匹配业务特点的接收策略，而不是所有串口都一个写法。
3. 控制层考虑了迟滞、最小切换间隔、报警延时和手自动共存，比较贴近真实工程控制。
4. 参数持久化做了 `magic + version + CRC32 + 去抖 + 跳过重复写入`，可靠性比较完整。
5. 远程控制不是简单收命令就执行，而是考虑了 `req_id` 幂等、ACK 缓存和延迟发送。

## 9. 面试时别说过头的地方

这部分很重要，能帮你避免被深追后接不住。

1. 不要说“我做了完整 MQTT 协议栈”。
   更准确的说法是：我实现了项目所需的 MQTT 3.1.1 基础报文封装和解析，覆盖连接、订阅、发布和保活。
2. 不要说“我写了通用 JSON 解析器”。
   更准确的说法是：我做了一个面向固定格式短 JSON 的轻量解析器。
3. 不要说“系统绝对不会丢包”。
   更准确的说法是：无线节点到网关采用 UDP，允许个别状态包丢失，但通过周期上报和在线超时机制保证整体状态可用。
4. 不要说“所有问题都靠 FreeRTOS 解决了”。
   更准确的说法是：FreeRTOS 解决了任务调度和资源管理问题，但协议边界、互斥、超时恢复和控制稳定性仍然需要自己设计。

## 10. 你最后可以这样收尾

如果面试官让你总结项目，你可以这样说：

这个项目让我比较完整地把嵌入式里常见的几类能力串起来了，包括 `多串口协议通信`、`FreeRTOS 任务设计`、`自动控制逻辑`、`本地屏交互`、`远程 MQTT 联动` 和 `Flash 参数持久化`。我觉得自己在这个项目里最大的收获不是单独某个模块能跑起来，而是知道怎么把多个模块整合成一个稳定的系统，并且在异常场景下还能可恢复、可调试、可维护。
